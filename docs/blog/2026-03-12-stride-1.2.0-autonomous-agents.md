# I Watched an AI Complete 10 Tasks in a Row Without Asking Me a Single Question

I typed "start claiming tasks" and walked away to make coffee.

When I came back, the agent had claimed a task, explored the codebase, written the implementation, run the full test suite, committed the code, and moved on to the next one. Then the next. Then the next. Ten tasks, zero interruptions, zero "what should I do next?" prompts.

This is Stride Plugin 1.2.0. And if you've ever watched an AI agent get stuck on a vague task description, choke on a goal that's too big, or spend twenty minutes trying to make sense of a wall of test failures — this release is for you.

For those new here: [Stride](https://www.stridelikeaboss.com) is a kanban-based task management platform built specifically for AI agents. You put tasks on a board, agents claim them, do the work, run validation hooks, and mark them done. The plugin gives agents the skills to do this properly — claiming tasks with the right setup, running tests before marking things complete, creating well-specified tasks. It's been working well. But agents still hit walls.

## Where Agents Used to Get Stuck

Three things would reliably break the flow.

First, **sparse tasks**. Someone writes "Add task commenting" with no other context. The agent claims it and immediately starts guessing. Which files? What patterns does the codebase use? What should the tests cover? It either asks the human for clarification (breaking the flow) or guesses wrong (wasting hours).

Second, **goals that are too big**. "Implement the hook diagnostician agent" sounds like one task, but it's actually four or five. The agent tries to do everything at once, loses track, and produces something that doesn't quite hold together.

Third, **cryptic failures**. The agent finishes its work, runs the test suite, and gets back a screenful of compilation errors, test failures, credo warnings, and formatting issues all jumbled together. It picks one to fix, but it's the wrong one — the compilation error was causing half the test failures, and fixing it first would have saved fifteen minutes.

Each of these was a moment where the agent would stall, ask for help, or waste time. In Stride 1.2.0, each one has a dedicated solution.

## The Complete Timeline: Every Skill and Agent in Action

Here's exactly what happens when an agent works through the Stride workflow, step by step. Every skill and agent is shown at the moment it's invoked.

---

### Step 1: Find Work

The agent calls `GET /api/tasks/next` and gets back a task from the Ready column.

**If it's a goal or a 25+ hour task:**

> **`stride-subagent-workflow`** checks the decision matrix and sees this needs decomposition.
>
> **`stride:task-decomposer`** (agent) is dispatched. It analyzes the goal's scope, identifies natural task boundaries — schema changes, business logic, UI, integration — maps dependencies between them, and estimates complexity for each piece. It returns an ordered list of child tasks, each fully specified.
>
> **`stride-creating-goals`** (skill) takes that list and creates the child tasks via `POST /api/tasks/batch` with proper dependency ordering. The goal now has claimable children.

The agent goes back to Step 1 and picks up the first child task.

---

### Step 2: Check Task Quality

The agent reviews the task's fields. Is it well-specified or sparse?

**If key_files, testing_strategy, or acceptance_criteria are missing:**

> **`stride-enriching-tasks`** (skill) kicks in. The agent explores the codebase in four phases: parse intent from the title, search for related files (schemas, contexts, LiveViews, tests), identify patterns used by similar features, and estimate complexity. It assembles a complete specification — key files with notes, acceptance criteria, testing strategy, verification steps, pitfalls, patterns to follow — and patches the task via the API.

What used to be a title on a card is now a detailed blueprint.

---

### Step 3: Set Up and Claim

> **`stride-claiming-tasks`** (skill) takes over. It verifies `.stride_auth.md` and `.stride.md` exist, reads the `before_doing` hook, and executes it:

```
git pull origin main          ← get latest code
mix deps.get                  ← install dependencies
mix ecto.migrate              ← run any new migrations
```

The hook result (exit code, output, duration) is captured. If it fails — merge conflict, missing dep, migration error — the agent fixes the issue and retries. No claiming until setup succeeds.

Once the hook passes, the agent calls `POST /api/tasks/claim` with the hook result. The task moves to Doing.

---

### Step 4: Explore and Plan

> **`stride-subagent-workflow`** (skill) checks the decision matrix again. For medium/large tasks or tasks with 2+ key_files:
>
> **`stride:task-explorer`** (agent) is dispatched. It reads every file in `key_files`, finds related test files, searches for patterns referenced in `patterns_to_follow`, navigates to `where_context`, and returns a structured summary: what exists, what patterns to follow, what utilities to reuse.
>
> **Plan agent** (built-in) is dispatched for medium+ tasks. It takes the explorer's output plus acceptance criteria, testing strategy, and pitfalls, and returns an ordered implementation plan.

For small tasks with 0-1 key_files, both are skipped — the agent codes directly.

---

### Step 5: Implement

The agent writes the code, following the plan (if one was generated), the patterns from the explorer, and the acceptance criteria from the task. This is the actual work — writing functions, updating templates, adding tests.

---

### Step 6: Pre-Completion Review

> **`stride-subagent-workflow`** (skill) checks the decision matrix one more time. For medium+ tasks or 2+ key_files:
>
> **`stride:task-reviewer`** (agent) is dispatched. It receives the full git diff plus the task's acceptance criteria, pitfalls, patterns_to_follow, and testing strategy. It checks whether the implementation actually meets the acceptance criteria, whether any pitfalls were violated, whether patterns were followed, and whether the testing strategy was honored. It returns either "Approved" or a list of issues categorized as Critical, Important, or Minor.

Critical and Important issues get fixed before proceeding.

---

### Step 7: Validate

> **`stride-completing-tasks`** (skill) takes over. It reads the `after_doing` hook from `.stride.md` and executes it:

```
mix test --cover              ← run tests with coverage
mix format --check-formatted  ← verify formatting
mix credo --strict            ← check code quality
mix sobelow --config          ← security scan
git commit -a -m "..."        ← commit the work
```

**If every command passes:** The hook result is captured and the agent moves to the next step.

**If something fails:**

> **`stride:hook-diagnostician`** (agent) is dispatched. It receives the raw output — a mix of compilation errors, test failures, credo warnings, and formatting issues — and returns a structured, prioritized fix plan.
>
> It parses six categories of failure, each with file paths, line numbers, and suggested fixes. More importantly, it orders them: fix compilation errors first (they're causing half the test failures), fix git conflicts before anything else (nothing works until the code merges), run the formatter last (it's a one-command fix).
>
> The agent follows the fix order, then re-runs the hook.

---

### Step 8: Prepare for Review

The `before_review` hook runs:

```
git fetch origin              ← get latest remote state
git rebase origin/main        ← rebase on top of main
mix test                      ← verify tests still pass after rebase
```

If it fails, the diagnostician analyzes the output again. Same pattern: structured diagnosis, prioritized fixes, retry.

---

### Step 9: Complete

Both hooks passed. The agent calls `PATCH /api/tasks/:id/complete` with both hook results, time spent, completion notes, and files changed.

The API responds with `needs_review`:

- **`needs_review: true`** — Task moves to Review. Agent stops and waits for a human to approve.
- **`needs_review: false`** — Task moves to Done. The `after_review` hook runs (fetch, rebase, test), and the agent **immediately goes back to Step 1** to claim the next task.

---

### The Loop

```
┌─ Step 1: GET /api/tasks/next
│    └─ Goal? → task-decomposer → stride-creating-goals → back to Step 1
│
├─ Step 2: Check task quality
│    └─ Sparse? → stride-enriching-tasks → patch task
│
├─ Step 3: stride-claiming-tasks
│    └─ before_doing hook → claim
│
├─ Step 4: stride-subagent-workflow
│    ├─ stride:task-explorer (if medium+ or 2+ key_files)
│    └─ Plan agent (if medium+ or 3+ key_files)
│
├─ Step 5: Implement
│
├─ Step 6: stride-subagent-workflow
│    └─ stride:task-reviewer (if medium+ or 2+ key_files)
│
├─ Step 7: stride-completing-tasks
│    ├─ after_doing hook
│    └─ Fails? → stride:hook-diagnostician → fix → retry
│
├─ Step 8: before_review hook
│    └─ Fails? → stride:hook-diagnostician → fix → retry
│
├─ Step 9: PATCH /api/tasks/:id/complete
│    ├─ needs_review: true → STOP
│    └─ needs_review: false → after_review hook
│
└─ Loop back to Step 1
```

No "what should I do?" No "can I call the API?" No stalling on vague descriptions. No floundering on failures.

Ten tasks. Zero interruptions. That's what Stride 1.2.0 feels like.

## Every Skill and Agent at a Glance

### Skills (workflow orchestration)

| Skill | When It's Invoked | What It Does |
|-------|-------------------|-------------|
| `stride-claiming-tasks` | Before claiming any task | Runs before_doing hook, captures result, calls claim API |
| `stride-completing-tasks` | After implementation is done | Runs after_doing + before_review hooks, calls complete API |
| `stride-creating-tasks` | When creating individual tasks | Enforces complete specification with all required fields |
| `stride-creating-goals` | When creating goals with child tasks | Handles batch creation with dependency ordering |
| `stride-enriching-tasks` | When a task is too sparse | Explores codebase to populate missing fields automatically |
| `stride-subagent-workflow` | After claim and before completion | Decides which agents to dispatch based on task complexity |

### Agents (specialized intelligence)

| Agent | When It's Dispatched | What It Returns |
|-------|---------------------|----------------|
| `stride:task-decomposer` | Goal or 25+ hour task detected | Ordered list of child tasks with dependencies and specs |
| `stride:task-explorer` | Medium+ task or 2+ key_files | Structured summary of key files, tests, patterns, and APIs |
| `stride:task-reviewer` | Before completion hooks | Categorized issues (Critical/Important/Minor) or "Approved" |
| `stride:hook-diagnostician` | When a blocking hook fails | Prioritized fix plan with file paths and suggested fixes |

# I Watched an AI Complete 10 Tasks in a Row Without Asking Me a Single Question

I typed "start claiming tasks" and walked away to make coffee.

When I came back, the agent had claimed a task, explored the codebase, written the implementation, run the full test suite, committed the code, and moved on to the next one. Then the next. Then the next. Ten tasks, zero interruptions, zero "what should I do next?" prompts.

This is Stride 1.2.0. And if you've ever watched an AI agent get stuck on a vague task description, choke on a goal that's too big, or spend twenty minutes trying to make sense of a wall of test failures — this release is for you.

For those new here: [Stride](https://www.stridelikeaboss.com) is a kanban-based task management platform built specifically for AI agents. You put tasks on a board, agents claim them, do the work, run validation hooks, and mark them done. The plugin gives agents the skills to do this properly — claiming tasks with the right setup, running tests before marking things complete, creating well-specified tasks. It's been working well. But agents still hit walls.

## Where Agents Used to Get Stuck

Three things would reliably break the flow.

First, **sparse tasks**. Someone writes "Add task commenting" with no other context. The agent claims it and immediately starts guessing. Which files? What patterns does the codebase use? What should the tests cover? It either asks the human for clarification (breaking the flow) or guesses wrong (wasting hours).

Second, **goals that are too big**. "Implement the hook diagnostician agent" sounds like one task, but it's actually four or five. The agent tries to do everything at once, loses track, and produces something that doesn't quite hold together.

Third, **cryptic failures**. The agent finishes its work, runs the test suite, and gets back a screenful of compilation errors, test failures, credo warnings, and formatting issues all jumbled together. It picks one to fix, but it's the wrong one — the compilation error was causing half the test failures, and fixing it first would have saved fifteen minutes.

Each of these was a moment where the agent would stall, ask for help, or waste time. In Stride 1.2.0, each one has a dedicated solution.

## The Enricher: Turning "Add Commenting" Into a Full Spec

The new `stride-enriching-tasks` skill kicks in when an agent encounters a task that's too sparse to implement confidently. Instead of guessing or asking the human, the agent explores the codebase in four phases.

It starts by parsing the intent from the title and description. Then it searches for related files — schemas, contexts, LiveViews, templates, existing tests. It looks at how similar features are built in the codebase to find patterns worth following. It estimates complexity. And then it assembles a complete task specification: key files with notes on why each matters, acceptance criteria, a testing strategy with unit and integration tests, verification steps with runnable commands, pitfalls to avoid, and patterns to follow.

What used to be a title on a card becomes a detailed blueprint. The agent doesn't need to ask "which files should I modify?" because it already figured that out. The task that would have taken three hours of exploration now takes thirty minutes of focused implementation.

## The Decomposer: Breaking Big Goals Into Ordered Work

The `stride:task-decomposer` agent handles the second problem. When the subagent workflow detects a goal, a task with large complexity, or anything estimated at 25+ hours, it dispatches the decomposer before anyone starts coding.

The decomposer analyzes the goal's scope, identifies natural task boundaries (schema changes, business logic, UI work, integration), maps dependencies between them (you can't build the UI for comments until the schema exists), and estimates complexity for each piece. The output is an ordered list of child tasks, each fully specified, ready to be created via the API and claimed individually.

This matters because agents work best on focused, well-scoped tasks. A three-hour task with clear boundaries and a known set of files is something an agent can nail on the first pass. A twenty-hour goal with vague edges is where things fall apart. The decomposer turns the latter into a series of the former.

During today's session, I watched the agent decompose a goal into five tasks, create them with proper dependencies, and then start claiming and completing them one by one. No confusion about scope. No trying to do too much at once.

## The Diagnostician: Making Sense of Failure

The `stride:hook-diagnostician` agent solves the third problem — and it might be my favorite of the three.

In Stride, agents run validation hooks after completing work: tests, linting, security scans, formatting checks. When everything passes, great. When something fails, the agent gets back a blob of mixed output from multiple tools and has to figure out what went wrong, what to fix first, and what might resolve itself once the root cause is addressed.

The diagnostician receives the raw hook output and returns a structured, prioritized fix plan. It parses six categories of failure — compilation errors, git conflicts, test failures, security warnings, credo issues, and formatting problems — each with specific file paths, line numbers, and suggested fixes. More importantly, it orders them correctly: fix compilation errors first because they're probably causing half the test failures. Fix git conflicts before anything else because nothing works until the code merges cleanly. Run the formatter last because it's a one-command fix.

This ordering is the key insight. An agent without the diagnostician might spend fifteen minutes fixing test assertions that would have passed on their own once the compilation error was resolved. The diagnostician prevents that wasted work by giving the agent the right sequence from the start.

## The Result

Back to my coffee. Here's what the workflow actually looks like now:

The agent calls `GET /api/tasks/next` and gets a task. If the task is sparse, the enricher fills it out. If it's a goal, the decomposer breaks it down. The agent runs the before_doing hook (pull latest code, install deps), claims the task, and starts implementing.

When it's done, it runs the after_doing hook. If the hook passes — tests green, credo clean, sobelow clear — it commits, runs the before_review hook, and calls the complete endpoint. If the hook fails, the diagnostician analyzes the output and the agent fixes issues in the right order.

Then it claims the next task. And the next. And the next.

No "what should I do?" No "can I call the API?" No stalling on vague descriptions. No floundering on failures. Just work.

Ten tasks. Zero interruptions. That's what Stride 1.2.0 feels like.

/plugin install stride@stride-marketplace

Already have it? Update to 1.2.0:

```
/plugin update stride
```
